# v0.2 Notes — Gaps Found During MVP Validation

The v0.1 MVP reached its acceptance bar: on 2026-07-07 a real toy project
(`greeter`) ran end-to-end on the real VMs with Claude Code as the worker —
packet → approve → code → checkpoint → deploy test → review → final report →
release approval → `main` — with no manual intervention beyond the two human
approvals. The full pipeline had zero integration failures.

These are gaps and deferrals observed along the way, to shape v0.2. None
block the MVP; all are deliberate scope boundaries or refinements.

## Wiring gaps

1. **The UI cannot start an autonomous run yet.** The dashboard Start button
   drives the run state machine and notifications only; the orchestrator loop
   is launched from the CLI (`loopwright run loop <project>`). v0.2 should
   spawn/supervise the loop as a background process from the UI (and reflect
   liveness), so a run can be started, watched, paused, and resumed entirely
   in the browser. Pause/resume/rollback and the release gate already work
   from the UI; only "start the loop process" is missing.

2. **No run supervisor/daemon.** `run_loop` runs in the foreground of whatever
   process invoked it. A long autonomous run needs a supervised background
   worker (systemd unit or a small daemon) so it survives terminal closure
   and the UI can attach to it.

## Behavior refinements

3. **The Developer VM is not reverted between sessions.** Only the Deployment
   VM reverts to a clean snapshot each cycle. The dev step uses a fresh git
   clone every time, but global VM state (installed packages, caches) persists
   across cycles. That is usually desirable (faster, warm toolchain) but means
   the dev environment is not provably reproducible. Consider an optional
   "revert dev VM to ready-for-run each session" mode.

4. **Reviewer does not yet check "docs touched when required."** The decision
   rules (`decision.py`) evaluate tests/deploy/tasks-remaining/failure but not
   whether documentation was updated when a task required it. Deferred until
   packets can declare per-task documentation requirements.

5. **Usage-limit reset time is not parsed.** On a usage-limit pause the loop
   auto-resumes after a fixed configurable delay (`limit_resume_minutes`).
   Claude's output often includes the actual reset timestamp; v0.2 could parse
   it and wait exactly that long instead of a fixed delay.

## Out of MVP scope (already planned as later features)

LangGraph orchestration, multi-agent worker pools, Playwright-based UI
acceptance testing, cost/token tracking, and richer diff-aware review.
